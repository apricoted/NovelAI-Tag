# 反馈系统上线指南

> 这份只讲「反馈问题」系统，不讲投稿后端。
> 你之前已经做完投稿后端的话，大部分 Cloudflare 配置都可以复用，不需要重做。

## 先说结论

反馈系统的到达链路是：

用户提交反馈 -> `POST /api/feedback` -> 写入 R2 -> `/review.html` 的「反馈」页查看和处理 -> 可选手机推送提醒。

如果投稿后端已经正常工作，你大概率已经有：

- `STRINGS_BUCKET`
- `ADMIN_TOKEN`

那么反馈系统必需配置基本已经齐了。你现在主要做的是：

1. 部署包含反馈代码的新版本。
2. 打开 `/review.html` 确认多出「反馈」页。
3. 可选配置 `SERVERCHAN_KEY`，让新反馈推送到手机。
4. 可选配置 `RATE_LIMIT_SALT` 和 WAF 速率限制，增强防刷。

## 不需要再做的事

这些是投稿系统需要的，反馈系统不需要你重新做：

- 不需要新建 R2 桶。
- 不需要重新开启 R2.dev 公开访问。
- 不需要重新配置 `STRINGS_PUBLIC_BASE`。
- 不需要配置 Turnstile。
- 不需要改 `submit.js` 的 Turnstile Site Key。
- 不需要新建单独的审核页。

反馈系统复用现有 `review.html`，只是多了一个「反馈」标签页。

## 必须确认的配置

### 1. 确认 R2 绑定还在

Cloudflare Dashboard：

1. 进入你的 Pages 项目。
2. 打开 `Settings` -> `Bindings`。
3. 确认存在一个 R2 bucket binding：

| 项 | 值 |
|---|---|
| Binding / Variable name | `STRINGS_BUCKET` |
| R2 bucket | 你之前投稿系统使用的桶 |

如果投稿系统现在能正常投稿和审核，这一项通常已经没问题。

### 2. 确认管理口令还在

Cloudflare Dashboard：

1. 进入 Pages 项目。
2. 打开 `Settings` -> `Variables and Secrets`。
3. 确认生产环境里有：

| 变量名 | 用途 |
|---|---|
| `ADMIN_TOKEN` | 打开 `/review.html` 的后台口令 |

如果你之前能进投稿审核页，这一项通常也已经没问题。

## 推荐新增的配置

### 1. `SERVERCHAN_KEY`（可选，但推荐）

用途：收到反馈后，给你手机推送一条提醒。

不配也没关系，反馈仍然会落到 R2，也能在 `/review.html` 查看。

配置位置：

Cloudflare Pages 项目 -> `Settings` -> `Variables and Secrets` -> Production -> Add variable

| 变量名 | 类型 | 值 |
|---|---|---|
| `SERVERCHAN_KEY` | Secret | 你的 ServerChan SendKey |

注意：

- 推送只是提醒，不是唯一收件箱。
- 就算推送失败，反馈也会先写入 R2，不会丢。
- 真正处理反馈还是去 `/review.html` 的「反馈」页。

### 2. `RATE_LIMIT_SALT`（可选，推荐）

用途：参与 IP hash 和软限流 key 生成，避免限流 hash 直接依赖 `ADMIN_TOKEN`。

配置位置同上：

| 变量名 | 类型 | 值 |
|---|---|---|
| `RATE_LIMIT_SALT` | Secret | 随机长字符串，建议 32 位以上 |

如果不配，代码会退回使用 `ADMIN_TOKEN`，再不行才使用固定 fallback。能用，但单独配一个更干净。

## 推荐加一条 WAF 速率限制

代码内已经有：

- 蜜罐字段
- 字段长度上限
- IP hash 软限流
- 待处理队列上限

但公开反馈接口没有登录态，也不使用 Turnstile，所以建议再用 Cloudflare WAF 做外层限速。

建议规则：

| 项 | 建议 |
|---|---|
| 匹配路径 | `/api/feedback` |
| 方法 | `POST` |
| 动作 | Block 或 Managed Challenge |
| 频率 | 可以先保守一点，例如同 IP 10 分钟 5-10 次 |

这项不是代码必需项，但线上公开后很值得加。

## 部署步骤

### 第 1 步：先部署包含反馈系统的新代码

这次代码里新增了这些关键文件：

- `functions/api/feedback.js`
- `functions/api/admin/feedback.js`
- `functions/api/admin/feedback-decide.js`
- `site/review.html` 的「反馈」后台页

所以必须等这批代码部署到 Cloudflare Pages 后，线上才会有 `/api/feedback` 和后台反馈页。

### 第 2 步：如果改了环境变量，重新部署一次

Cloudflare Pages 的绑定和环境变量变更，需要新部署后才稳定生效。

如果你只是确认已有 `STRINGS_BUCKET` / `ADMIN_TOKEN`，没有新增变量，正常发布新代码即可。

如果你新增了 `SERVERCHAN_KEY` 或 `RATE_LIMIT_SALT`，建议保存后再部署一次。

### 第 3 步：线上测试反馈提交

打开你的正式站点：

1. 打开主站。
2. 点「更多」里的「反馈问题」。
3. 类型随便选一个。
4. 描述填一条测试内容，例如：

```text
这是反馈系统上线测试，看到后可以标记已处理。
```

5. 提交。

成功时，用户侧会看到提交成功提示。

### 第 4 步：后台确认收到

打开：

```text
https://你的域名/review.html
```

然后：

1. 输入原来的 `ADMIN_TOKEN`。
2. 切到「反馈」。
3. 默认会看到「待处理」。
4. 找到刚刚那条测试反馈。
5. 可以点「复制上下文」确认自动打包信息正常。
6. 点「标记已处理」。
7. 切到「已处理」，确认记录已经移动过去。

## R2 里会长什么样

未处理反馈：

```text
feedback/pending/YYYY/MM/<id>.json
```

标记已处理后：

```text
feedback/resolved/YYYY/MM/<id>.json
```

忽略后：

```text
feedback/ignored/YYYY/MM/<id>.json
```

限流记录：

```text
feedback/ratelimit/<ipHash>/<window>.json
```

不会保存明文 IP，只保存 hash。

## 本地测试方式

普通 `启动预览.bat` 是静态预览，不会运行 Pages Functions，所以不能完整测试 `/api/feedback`。

要测反馈接口，用：

```text
投稿本地测试.bat
```

它会启动：

```text
http://localhost:8788/
```

本地后台：

```text
http://localhost:8788/review.html
```

本地口令：

```text
devtoken
```

这个脚本会通过 Wrangler 启动 Pages Functions，并挂上本地模拟的 `STRINGS_BUCKET`。

## 常见问题

### 提交反馈后 404

通常说明线上部署的版本还没有包含 `functions/api/feedback.js`。

处理：

- 确认代码已经合并 / 推送。
- 确认 Cloudflare Pages 已重新部署。

### 提交反馈提示未绑定 `STRINGS_BUCKET`

说明 Pages Functions 没拿到 R2 binding。

处理：

- Cloudflare Pages -> `Settings` -> `Bindings`
- 确认 R2 binding 名字是 `STRINGS_BUCKET`
- 保存后重新部署。

### `/review.html` 口令错误

说明 `ADMIN_TOKEN` 不对或环境变量未生效。

处理：

- 检查 Production 环境变量 `ADMIN_TOKEN`
- 确认输入的是同一串
- 改完后重新部署。

### 后台没有看到反馈，但用户显示提交成功

先看 R2：

- 有没有 `feedback/pending/`
- 有没有 `feedback/resolved/` 或 `feedback/ignored/`

如果 R2 有，但后台没有：

- 确认打开的是新版本 `/review.html`
- 刷新页面
- 确认选中的是「待处理」而不是「已处理」或「已忽略」

### 手机没有推送

这不影响反馈落库。

检查：

- `SERVERCHAN_KEY` 是否正确
- 是否配置在 Production
- 是否重新部署
- ServerChan 本身是否可用

后台 `/review.html` 能看到反馈，就说明核心链路正常。

## 最小上线清单

如果你不想折腾，按这个做就够：

1. 合并并部署包含反馈系统的新代码。
2. 确认旧的 `STRINGS_BUCKET` 还在。
3. 确认旧的 `ADMIN_TOKEN` 还在。
4. 打开主站提交一条测试反馈。
5. 打开 `/review.html` -> 「反馈」确认能看到。
6. 可选：补 `SERVERCHAN_KEY` 手机提醒。
7. 可选：补 WAF 限速。

到第 5 步成功，反馈系统就已经能用了。

## 参考

- Cloudflare Pages Functions 通过 bindings 访问 R2，绑定名会出现在 `context.env`。
- R2 binding 修改后需要重新部署 Pages 才会生效。
- 本地测试 R2 binding 可以用 `wrangler pages dev site --r2 STRINGS_BUCKET`。
- 后台推送使用 `context.waitUntil()`，所以推送失败不会阻断反馈写入。

官方文档：

- https://developers.cloudflare.com/pages/functions/bindings/
- https://developers.cloudflare.com/workers/runtime-apis/context/#waituntil
